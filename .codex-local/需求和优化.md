# 需求和优化

## 背景

- 主题：运营管理 -> 课程详情 -> 追问数据页
- 当前分支：`feat/operator-course-followup-page`
- 当前重点：
  - 先保证追问详情中的 source 回溯尽量符合历史事实
  - 把后续性能优化和继续排查项沉淀下来，方便后面定期回捞

## 状态说明

- `已完成`：当前分支已经落地，后续只需回归验证或观察
- `部分完成`：已经处理了一部分，但仍有残留范围需要继续统一
- `待处理`：尚未开始，后续回捞时需要优先确认是否仍有价值

## 已落地修复

### 1. 追问详情的 element source 改为按追问时间回溯历史快照

- 状态：`已完成`

- 问题：之前直接读取最新 active 的 anchor element，历史追问可能读到后来 reload 后的新内容
- 当前口径：
  - 已知 `anchor_element_bid`
  - 按 `created_at <= 当前追问时间` 回查最近一条未删除的 element 快照
  - 不再依赖“当前是否 active”判断历史 source

### 2. 追问详情的 block fallback 改为允许命中历史 superseded teacher block

- 状态：`已完成`

- 问题：之前 fallback 只查 `status == 1` 的 teacher block，历史上被 supersede 的原始输出会查不到
- 当前口径：
  - 同 `progress_record_bid`
  - 同 `position`
  - `role = teacher`
  - `type in (MDINTERACTION, MDCONTENT)`
  - 时间早于当前 ask
  - 允许命中历史行，不再只看 active 行

### 3. 翻页组件位置与运营其他页面保持一致

- 状态：`已完成`

- 已处理：
  - 追问数据页分页组件已改为与运营其他列表页一致的靠右对齐方式
  - 对齐参数与课程详情页、运营用户页保持一致

### 4. 无章结构课程时，列表与筛选文案按“节”展示

- 状态：`部分完成`

- 已处理：
  - 当当前列表数据不存在真实章 / 节两层结构时：
    - 列表列名从“章节”切换为“节”
    - 筛选文案从“章/节搜索”切换为“节搜索”
    - placeholder 同步切换为“输入节标题”
- 仍待继续统一：
  - 详情字段名
  - 相关空态或提示文案
  - 其他后续可能新增的追问页派生视图

## 页面产品优化项

### 总体判断

- 当前页面的大结构合理：
  - 概览卡片
  - 筛选区
  - 列表区
  - 详情抽屉
- 当前页面的主要问题不在大框架，而在语义容易被误解：
  - source 缺失态表达过弱
  - 追问轮次定义不够清晰
  - 章 / 节字段在不同课程结构下容易让运营困惑
  - 详情中“原始输出 / 本次追问 / 系统回答”的先后关系不够一眼看懂

### 必须改

#### 1. source 缺失态要明确表达

- 状态：`已完成`

- 当前问题：
  - 直接显示 `--` 信息量过低
  - 运营无法区分“确实没有”还是“历史数据暂时无法定位”
- 建议：
  - 改为：
    - `暂无法定位对应原始输出`
    - 或 `历史数据未记录对应原始输出`
- 目标：
  - 降低误解
  - 避免用户把 source 缺失理解为系统异常或空数据

#### 2. 明确“追问轮次”的定义

- 状态：`已完成`

- 当前问题：
  - `第 N 次` 很容易被理解成“对同一条原始输出的第 N 次连续追问”
  - 但当前实现实际是“同一学习记录中的追问时间顺序”
- 建议：
  - 在列表列头或详情字段旁增加 tooltip / 说明文案
  - 文案建议：
    - `按同一学习记录中的追问时间顺序排序`
- 目标：
  - 让运营正确理解轮次口径

#### 3. 章节 / 节语义继续统一

- 状态：`部分完成`

- 当前问题：
  - 有的课程存在章 + 节两层结构
  - 有的课程只有节，没有章
  - 如果页面文案写死“章/节”或“章节”，会造成理解偏差
- 建议：
  - 继续统一以下位置的口径：
    - 列表列名
    - 筛选文案
    - 详情字段名
    - 空态文案
  - 当课程无章结构时，优先统一显示为“节”
- 目标：
  - 保持整页语义一致
- 备注：
  - 当前分支已完成列表列名和筛选文案的动态切换
  - 详情字段名和其他文案仍需后续补齐

#### 4. 详情中三类内容的关系要更清楚

- 状态：`已完成`

- 当前问题：
  - 用户需要自己推断：
    - 被追问的原始输出
    - 当前追问
    - 系统回答
  - 尤其 source 缺失时，更容易困惑
- 建议：
  - 在标题和布局顺序上更明确地区分这三者
  - 推荐顺序：
    - `被追问的原始输出`
    - `本次追问`
    - `系统回答`
- 目标：
  - 让运营快速理解“因 -> 问 -> 答”的关系

## 运营课程详情页 / 积分消耗页优化记录

## 背景

- 主题：运营管理 -> 课程管理 -> 课程详情 -> 积分消耗
- 当前分支：`feat/operator-course-credit-page`
- 当前问题：
  - 当前列表按单条 `usage` 明细展示
  - 真实计费并没有问题，但运营查看时会看到大量同手机号、同小节、同时间段的重复行
  - 页面信息过细，更像底层结算流水，而不是运营需要的“这次学习大概消耗了多少”

## 当前结论

- 不调整积分计费口径
- 不改后端 settlement 逻辑
- 只调整运营查看口径
- 不建议只做前端聚合：
  - 现有接口先按明细分页，再回前端渲染
  - 如果前端再聚合，会出现跨页拆组、总数失真、排序失真的问题
- 建议做法：
  - 后端提供聚合视图
  - 前端默认展示聚合结果
  - 后续如有必要，再补原始明细展开能力

## 分阶段方案

### A. 默认聚合视图（本次实施）

- 状态：`已完成`
- 本次实现范围：

  - B 只做需求记录，不进入本次实现范围

    - 只允许合并“同一进度、同一场景”下的多条记录
  - fallback：
    - 如果某条记录没有 `progress_record_bid`
    - 不强行合并，继续按单条展示，避免误归组

- 后端要求：
  - 在课程积分消耗接口上支持 grouped 视图
  - grouped 视图必须在后端完成分页、总数和排序
  - 不接受“后端仍按 raw 明细分页，前端再二次合并”的做法

    - `raw` 口径先保留给后续 B 使用，不要求本次在页面开放切换入口

- grouped 排序和分页口径：
  - 默认按“最近消耗时间倒序”
  - 分页基于 grouped 结果本身计算
  - total/page_count 都对应 grouped 后的组数，不对应 raw usage 行数

  - `group_key`

  - 章节
  - 小节
  - 总消耗积分
  - 模型摘要

- grouped 字段规则：
  - 最近消耗时间：组内 `created_at max`
  - 总消耗积分：组内 `consumed_credits sum`

  - 模型摘要：
    - 单模型则直接显示
    - 多模型则显示压缩摘要，如“qwen / volcengine 等 N 个”

### C. 课程积分消耗列表的 SQL 级分页 / 聚合优化（后续迭代补充）

- 状态：`待处理`
- 当前问题：
  - 现有实现会先执行 `query.order_by(...).all()`
  - 再在 Python 内存里做 grouped 聚合和分页
  - 当单课程 usage 记录很多时，接口会有明显的内存和响应时间压力
  - 当前实现更偏“先保证运营可用”，还不是最终性能形态

- 当前问题边界：
  - 这不是计费准确性问题
  - 这不是当前页面展示错误问题
  - 这是大数据量下的性能 / 稳定性优化项
  - 本轮不继续扩大改造范围，后续单独迭代

- 后续优化目标：
  - raw 视图改成数据库层分页，不再全量拉取后再切页
  - grouped 视图改成数据库层聚合 + 分页，不再把所有 raw 行先加载到 Python
  - 保持当前 grouped 展示口径不变：
    - 主分组键仍为 `progress_record_bid + usage_mode`
    - 无 `progress_record_bid` 的记录继续按单条兜底

- 推荐优化方案：
  - raw 视图：
    - 对基础查询先做 `count`
    - 再执行 `order by + offset + limit`
    - 只取当前页 raw 行
  - grouped 视图：
    - 在 SQL 层先产出 grouped key
    - grouped key 口径：
      - 有 `progress_record_bid` 时，按 `progress_record_bid + usage_mode`
      - 无 `progress_record_bid` 时，按 `usage_bid`
    - 在 SQL 层完成：
      - `count(*) -> usage_count`
      - `sum(consumed credits) -> total_consumed_credits`
      - `max(created_at) -> latest_created_at`
      - 当前组对应的主展示记录选择
      - provider/model 摘要所需的去重信息
    - 对 grouped 结果本身再做 `order by + offset + limit`
    - 最后只回填当前页所需的 user / outline 展示信息

- 实施建议：
  - 优先先改 raw 视图分页下沉，风险更低

- 即使用户当前无有效套餐，只要人工发放积分仍在有效期内，仍可消费

#### 3. 积分包新增规则（已纳入本次方案前提）

- 积分包默认永久有效
- 但只有“存在有效套餐”时才可以消费积分包
- 套餐失效后：
  - 积分包余额保留
  - 但进入不可消费状态
- 后续重新开通有效套餐后：
  - 积分包恢复可消费

#### 1. 发放套餐与购买套餐的“仅支持升级”规则收敛为同一套校验

- 预期收益：
  - 购买套餐与发放套餐的升级约束长期保持一致
  - 后续购买套餐改规则时，运营发放套餐不需要再额外人工同步维护

#### 2. 运营发放套餐的后端拒绝判断继续细分，避免错误文案过泛或过窄

- 当前现状：
  - `运营发放套餐` 在 `billing/manual_plan_grants.py` 内，当前存在一条较早的拒绝分支：
    - 当前有效套餐若不是 `manual` provider，会直接报错 `adminPlanGrantProviderManagedConflict`
  - 这会把多种不同场景合并成同一条错误：
    - 当前已有更高等级有效套餐，发放更低等级套餐
    - 当前已有支付中的有效套餐，但后台这条路径本身不允许直接覆盖
  - 结果是前端只能拿到单一错误码，文案很难既准确覆盖“高等级拦截”场景，又不误伤其他 provider-managed 场景
- 后续目标：
  - 在后端补充更细的判断分支，而不是继续复用同一条冲突错误
  - 至少区分：
    - 当前已有更高等级有效套餐，禁止发放低等级套餐
    - 当前已有支付中的有效套餐，但该 provider 不允许后台直接发放覆盖
  - 需要时可继续细分：
    - 同级套餐拦截
    - 同套餐重复发放拦截
- 建议实施方向：
  - 先统一读取“当前有效套餐 + 当前 provider + 当前套餐 sort_order + 目标套餐 sort_order”
  - 再按明确优先级输出更精确的后端错误码
  - 前端继续只消费后端返回的业务文案，不在前端猜测套餐等级关系
- 预期收益：
  - 运营看到的错误提示更贴合真实限制原因
  - 后续如果继续扩展“后台发放套餐”的限制规则，文案与判定可以保持一一对应

#### 3. 运营发放套餐的幂等兜底从 Redis 锁补强到数据库层

- 当前现状：
  - `运营发放套餐` 的幂等入口在 `billing/manual_plan_grants.py`
  - 当前主路径依赖 `cache_provider.cache.lock(...)` / Redis 锁避免同一 `request_id` 并发重复发放
  - 如果 Redis 不可用或退化到进程内锁，多 worker / 多实例场景下仍然可能出现重复写单
  - 目前 `bill_orders.provider_reference_id` 只有普通索引，没有针对“人工发放套餐幂等键”的唯一约束
- 后续目标：
  - 将“后台发放套餐”的最终幂等保证下沉到数据库层，而不是只依赖缓存锁
  - 即使 Redis 异常、锁失效或并发请求落到不同 worker，也能稳定回收为同一笔结果
- 建议实施方向：
  - 为人工发放套餐这条路径设计明确的数据库唯一键
    - 可选方向：围绕 `creator_bid + payment_provider + provider_reference_id`
    - 或单独抽出专用幂等字段 / 表，避免和历史订单语义混用
  - 发放时改为：
    - 先按统一幂等键尝试写入
    - 遇到唯一冲突后回查既有订单
    - 返回既有结果，而不是继续创建第二笔 paid order
  - Redis 锁可以保留，作为减压与降低冲突概率的前置优化，但不再是唯一正确性保障
- 实施边界：
  - 这是后端持久化与并发语义增强，不建议夹带在当前前端 / 运营体验类 PR 内顺手改
  - 如果引入唯一约束，需要配套 migration、历史数据自查与冲突回放验证
- 预期收益：
  - 后台发放套餐在多实例并发下的幂等更稳
  - 后续 review 不需要再反复解释“当前主要靠 Redis 锁是否足够”
  - 运维层 Redis 异常不会直接放大为重复发放风险

#### 4. billing 订单失败 / 超时口径与回收机制改造

- 当前现状：
  - billing 域里虽然定义了 `timeout` 订单状态，但当前代码里没有查到一条明确的“本地超时扫描任务 / 固定待支付 N 分钟后自动转 timeout”链路
  - 现阶段更接近：
    - 订单创建后先进入 `pending`
    - 后续依赖 provider webhook / sync 推进到 `paid` / `failed` / `canceled`
  - 典型失败来源：
    - Stripe `payment_intent.payment_failed`
    - Stripe checkout session `expired`
    - subscription renewal 对应的 provider 状态进入 `past_due` / `unpaid` / `incomplete_expired` / `canceled`
    - native provider 回传失败态
  - 当前也允许：
    - 订单先进入 `failed`
    - 后续若 webhook / sync 证明已支付成功，再回正为 `paid`
  - 支付成功后已有短信 / 飞书通知链路
  - 支付失败当前没有单独的失败通知链路，主要依赖 admin 订单页 / 异常页 + provider raw payload 排查
- 对套餐活动的直接影响：
  - 如果不先明确 pending 单的有效期与锁价边界，后续套餐活动会出现“活动结束后，老 pending 单是否还保留活动价”的口径不清问题
  - 当前更稳妥的方向是：
    - 创建订单时锁定当时活动快照与最终 payable amount
    - 后续只要这笔订单仍然有效，就按订单快照支付，不重新按当前活动配置回算
- 后续目标：
  - 明确 billing 订单在运营与产品层的统一口径：
    - `pending`：待支付中的 provider 订单
    - `failed`：provider 明确失败或 sync 识别失败
    - `canceled`：provider / 用户明确取消
    - `timeout`：是否启用本地超时态，如果启用，具体超时分钟数与落库时机是什么
  - 明确套餐活动对老 pending 单的价格策略：
    - V1 建议按“下单锁价”执行
    - 若订单失效后需要重新支付，应重新创建订单，再按当下活动重新计算
- 建议实施方向：
  - 在套餐活动正式开发前，先把“锁价规则 + 失效重建规则 + timeout 是否启用”写成明确产品约束
  - 若后续需要补 timeout：
    - 再单开迭代补 billing 本地超时任务 / provider 过期回收机制
  - admin 侧排查口径继续统一为：
    - 先看 `bill_orders.failure_message / failure_code`
    - 再看 `bill_orders.metadata.latest_provider_payload`
    - 最后再看 raw snapshot 表与应用日志
- 预期收益：
  - 套餐活动落地时不会因为订单状态口径不清导致价格回算歧义
  - 运营与测试都能更稳定判断“这笔活动单为什么是这个价格”

#### 5. 课程基础定价按站点 / 支付渠道配置化

- 记录时间：2026-06-02
- 当前分支：`fix/course-base-pricing`
- 当前状态：`已落地`
- 背景：
  - 国际站接入 Stripe 后，Stripe 对美元订单存在最低扣款金额限制，课程价格低于该限制时调起 Stripe 收银台有失败风险。
  - 国内站不一定需要沿用 Stripe 的 `0.5` 最低价格门槛；如果国内站只走支付宝 / 微信 / Ping++ 等国内支付渠道，课程基础定价不应被 Stripe 规则误限制。
  - 当前课程价格最低值在前后端都有 `0.5` 口径：
    - 前端：`src/cook-web/src/components/shifu-setting/ShifuSetting.tsx`
    - 后端配置默认值：`src/api/flaskr/common/config.py`
    - 后端保存校验：`src/api/flaskr/service/shifu/shifu_draft_funcs.py`
- 当前风险：
  - 前端硬编码最低价格，不能按 CN / COM 或支付渠道配置灵活展示。
  - 后端虽然有 `MIN_SHIFU_PRICE` 配置，但 runtime-config 未透出给前端，导致前后端口径可能不一致。
  - C 端支付如果误选 Stripe 且金额低于 Stripe 最低扣款额，当前后端可能直接调用 Stripe，失败后容易表现为“系统异常”。
  - `payment_channel` 默认解析如果同时启用多个支付渠道，存在优先走 Stripe 的可能；国内低价课程必须避免国内环境误启用 Stripe。
- 推荐方案：
  - 用站点 / 支付渠道配置控制课程最低价格，而不是用语言或前端写死判断。
  - 国内站建议配置：
    - `MIN_SHIFU_PRICE=0` 或按业务需要配置为 `0.01`
    - `PAYMENT_CHANNELS_ENABLED=alipay,wechatpay` 或 `pingxx`
    - `STRIPE_ENABLED=false`
  - 国际站建议配置：
    - `MIN_SHIFU_PRICE=0.5`
    - `STRIPE_MIN_CHARGE_AMOUNT=50`
    - `PAYMENT_CHANNELS_ENABLED=stripe`
    - `STRIPE_ENABLED=true`
    - `STRIPE_DEFAULT_CURRENCY=usd`
  - 后端 runtime-config 增加 `minShifuPrice`，前端课程设置页使用该值做展示与校验。
  - 后端 Stripe 支付前增加最低金额兜底校验：当 `payment_channel=stripe` 且金额大于 0 但低于 Stripe 最低金额时，返回明确业务错误，不再让 provider 异常冒泡成“系统异常”。
  - 后端创建课程草稿时，`shifu_price=0` 需要被视为合法显式值，不应被 `or MIN_SHIFU_PRICE` 误替换。
- C 端支付影响判断：
  - 国内站如果禁用 Stripe 且只展示国内支付渠道，C 端支付流程不应受 Stripe 最低金额影响。
  - 国际站仍保留 Stripe 最低金额口径，避免低价课程进入 Stripe 后失败。
  - 如果环境配置错误导致国内站仍启用 Stripe，后端兜底会返回明确错误，避免调起收银台后才报“系统异常”。
- 建议实施清单：
  - 后端：
    - `RuntimeConfigDTO` 补 `minShifuPrice`
    - `/api/runtime-config` 返回 `MIN_SHIFU_PRICE`
    - 课程草稿保存继续用后端配置做最终校验
    - 订单支付链路对 Stripe 低金额增加业务错误兜底
  - 前端：
    - env store / runtime 初始化补 `minShifuPrice`
    - 课程设置页移除 `0.5` 硬编码，改用 runtime-config
    - 错误提示文案走 i18n，按当前配置展示最低价格
  - 配置：
    - 国内站和国际站分别确认 `MIN_SHIFU_PRICE`、`PAYMENT_CHANNELS_ENABLED`、`STRIPE_ENABLED`、`STRIPE_MIN_CHARGE_AMOUNT` 的生产配置。
- 回归测试点：
  - 国内站配置 `MIN_SHIFU_PRICE=0` 时，后台创建 / 修改课程价格允许 `0` 或国内业务允许的最低值。
  - 国际站配置 `MIN_SHIFU_PRICE=0.5` 时，低于 `0.5` 的课程价格仍被拦截。
  - C 端国内支付渠道下单不受 Stripe 最低金额影响。
  - C 端 Stripe 低金额订单返回明确业务提示，不出现“系统异常”。
  - runtime-config 与前端课程设置页展示的最低价一致。

#### 7. 后台个人中心密码能力补全（初步方案，后续迭代）

- 记录时间：2026-06-09
- 当前现状：
  - 后台左下角用户菜单已经补了“设置密码”入口，但当前入口复用的是首次设置密码弹窗。
  - 前端现有弹窗走的是 `POST /user/set_password`，适用于“已登录 + 手机/邮箱验证码 + 首次设置密码”。
  - 后端实际已经有三条密码链路：
    - `POST /user/set_password`
      - 首次设置密码
      - 如果密码已存在，会返回 `server.user.passwordAlreadySet`
    - `POST /user/change_password`
      - 已登录用户修改密码
      - 需要 `old_password + new_password`
    - `POST /user/reset_password`
      - 忘记密码场景
      - 通过验证码重置
- 当前问题：
  - 后台“设置密码”入口语义不完整。
  - 用户首次设置成功后，如果再次进入该入口，继续走 `set_password` 会被后端拒绝。
  - 也就是说，现在后台入口本质上只是“首次设置密码”，不是完整的“密码管理”。
- 建议目标：
  - 后台个人中心密码入口按用户状态区分：
    - 未设置过密码：显示“设置密码”
    - 已设置过密码：显示“修改密码”
  - 不把“修改密码”和“忘记密码”混成一套流程。
  - 后台个人中心只承载登录态内的密码管理，不承担找回密码。
- 建议方案：
  - 后端：
    - 增加一个轻量状态字段，供前端判断当前用户是否已设置密码。
    - 优先方案：
      - 在 `/api/user/info` 或 `/api/runtime-config` 关联的用户信息链路里补一个布尔字段，例如：
        - `has_password`
    - 该字段只表示当前账号是否存在有效 password credential，不返回任何敏感信息。
  - 前端：
    - 后台左下角用户菜单入口改成状态化文案：
      - `has_password=false`：显示“设置密码”
      - `has_password=true`：显示“修改密码”
    - 保留现有 `SetPasswordModal` 作为首次设置弹窗。
    - 新增一个 `ChangePasswordModal`：
      - 字段：
        - `old_password`
        - `new_password`
        - `confirm_password`
      - 提交接口：
        - `POST /user/change_password`
      - 校验口径继续复用现有密码强度规则。
  - 交互：
    - 首次设置成功后，刷新 `userInfo`，让入口文案自动切成“修改密码”。
    - 修改密码成功后，仅提示成功并关闭弹窗，不需要重新登录。
- 不建议的方向：
  - 不建议让已设置密码的用户继续走验证码 + `set_password` 这条首次设置链路。
  - 不建议在后台个人中心里直接塞“忘记密码”流程。
  - 不建议把“设置密码”和“修改密码”硬合成一个弹窗，用接口报错来区分状态。
- 最小实施顺序建议：
  - 第一步：
    - 后端补 `has_password`
    - 前端入口文案按状态切换
  - 第二步：
    - 新增 `ChangePasswordModal`
    - 接上 `/user/change_password`
  - 第三步：
    - 学习端与后台统一密码管理入口语义，避免一边叫“设置密码”、一边实际走“修改密码”
- 回归测试点：
  - 未设置密码账号进入后台菜单，显示“设置密码”，提交成功后可正常密码登录。
  - 已设置密码账号进入后台菜单，显示“修改密码”，不再调用 `set_password`。
  - 修改密码时旧密码错误，返回明确错误提示。
  - 修改密码成功后，新密码可登录，旧密码失效。
  - 没有手机号/邮箱的账号，后台不展示首次设置密码入口，或保持现有无联系方式兜底。
