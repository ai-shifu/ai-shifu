# Billing Credit Notifications Requirements

Updated: 2026-05-20

## Goal

Define one requirements surface for creator credit notifications so credit
reminders are not scattered across auth SMS, manual grant, wallet query, or
settlement paths.

The v1 delivery channel is SMS. The product model should still be a
notification center so future channels such as in-app messages, email, or
Feishu can reuse the same trigger, dedupe, state, and operator review rules.

This document is requirements only. It does not introduce API, schema, task,
configuration, or frontend type changes by itself.

## Existing References

- `docs/billing-subscription-purchase-sms.md` defines the current asynchronous
  billing SMS pattern: write notification intent with the business truth, enqueue
  after commit, and let a worker send through the provider.
- `docs/operator-user-points-grant.md` defines manual credit grant semantics for
  reward and compensation credits.
- `docs/billing-subscription-design.md` defines wallet, bucket, ledger, bucket
  expiration, and low-balance alert boundaries.

## Scenarios

### `credit_expiring`

Notify a creator before active credits expire.

- Source of truth: `credit_wallet_buckets`.
- Source id: `wallet_bucket_bid`.
- Trigger: scheduled scan over active buckets whose `effective_to` falls within
  configured reminder windows.
- Suggested v1 windows: `7d`, `3d`, `1d`, and `0d`.
- Dedupe key: `credit_expiring:{wallet_bucket_bid}:{window}`.
- Content variables: expiring credits, expiry time, and recharge or subscription
  action entry.
- Multiple buckets for one creator and one reminder window should be merged into
  one user-facing notification when possible.

### `credit_granted`

Notify a creator after credits are successfully granted.

- Source of truth: `credit_ledger_entries`.
- Source id: `ledger_bid`.
- Trigger: event after the grant ledger and matching bucket are committed.
- Covered grant sources: trial, paid subscription/topup grant, manual reward,
  manual compensation, and future campaign grants.
- Dedupe key: `credit_granted:{ledger_bid}`.
- Content variables: granted credits, grant source label, expiry time when
  available, and credit detail or billing entry.
- Duplicate grant requests that reuse an existing ledger entry must not send a
  second reminder.

### `low_balance`

Notify a creator when usable credits are low.

- Source of truth: `credit_wallets` plus consumable bucket state.
- Source id: `creator_bid`.
- Trigger: scheduled scan over creator wallets and billing overview alerts.
- Current billing v1 low-balance alert uses `available_credits <=
  BILL_LOW_BALANCE_THRESHOLD`, and the current bootstrap threshold is `0`.
- Future product thresholds may add fixed credit amount, percentage, or estimated
  days remaining, but those thresholds must be explicitly configured.
- Dedupe key: `low_balance:{creator_bid}:{threshold}:{date}`.
- Content variables: current available credits, threshold label, and checkout or
  subscription action entry.

## Trigger Rules

- Event-triggered notifications are created only after the business state commits.
  For v1 this mainly applies to `credit_granted`.
- Scan-triggered notifications are generated by scheduled worker tasks. For v1
  this applies to `credit_expiring` and `low_balance`.
- Route handlers must not send SMS directly.
- Payment provider adapters, auth verification-code helpers, and wallet read
  APIs must not own credit notification business rules.
- Provider failures may be retried. Skipped states caused by no mobile, opt-out,
  blacklist, or duplicate suppression are terminal unless operator policy changes.

## Recipient Rules

- The default recipient is the creator whose credits are granted, expiring, or
  low.
- Mobile number is resolved from the creator user account profile.
- Phone identifiers must use the same normalized phone storage rules as auth and
  user lookup flows.
- If the creator has no usable mobile number, mark the notification as
  `skipped_no_mobile`.
- If the creator has opted out, is in a blacklist, or is blocked by operator
  policy, mark the notification as skipped and do not call the SMS provider.
- Internationalization and timezone formatting must use the user's language and
  the application timezone where available.

## Notification State

Each notification attempt should have a durable state record or equivalent
metadata entry with at least:

- notification type: `credit_expiring`, `credit_granted`, or `low_balance`
- channel: v1 uses `sms`
- creator BID and target user BID
- mobile snapshot used for delivery
- source type and source id
- dedupe key
- status
- requested, sent, and updated timestamps
- provider response summary and error code when applicable

Required statuses:

- `pending`: notification intent exists and can be processed
- `sent`: provider accepted the SMS request
- `skipped_no_mobile`: no usable recipient mobile was available
- `skipped_opt_out`: recipient or operator policy blocks delivery
- `suppressed_duplicate`: the dedupe key already has a processed notification
- `failed_provider`: SMS provider failed and the notification may be retried

## Operator Requirements

Operators need a read-only management surface for v1:

- List notification records by creator, mobile, notification type, channel,
  status, source id, and time range.
- Inspect provider response and error code for failed notifications.
- Requeue notifications in `failed_provider` status.
- See why a notification was skipped.
- Configure per-type enablement, SMS template binding, reminder windows, and
  low-balance thresholds.
- Run a dry-run count before enabling a scan-based notification policy.

Operator actions must not mutate wallet, bucket, or ledger facts. Requeue only
changes notification delivery state.

## Metrics

Track product and operational metrics by notification type and channel:

- generated notifications
- sent notifications
- provider failures
- skipped notifications by reason
- duplicate suppressions
- SMS cost estimate
- post-notification conversion, such as login, checkout, topup, subscription
  renewal, course creation, or continued credit consumption

Metrics are reporting data only. Credit balances and consumption truth remain in
the billing ledger and bucket tables.

## Boundaries

- Credit notifications do not grant, consume, expire, or adjust credits.
- Credit truth remains in `credit_ledger_entries`, `credit_wallet_buckets`, and
  `credit_wallets`.
- Bucket expiration remains owned by `billing.expire_wallet_buckets`.
- Low-balance alert detection remains a billing concern and should not add new
  runtime admission error codes.
- Do not reuse `/api/user/send_sms_code`, `/api/user/console_send_sms_code`, or
  verification-code templates for credit notifications.
- Do not send notification SMS before the source billing fact is committed.

## Acceptance Criteria

- The three v1 notification types are clearly defined with trigger source,
  source id, and dedupe key.
- The document separates notification delivery from credit ledger mutation.
- The document keeps auth SMS and credit notification SMS separate.
- The document defines skip, failure, duplicate, and retry behavior at the
  requirements level.
- The document references existing billing SMS, manual grant, expiration, and
  low-balance design surfaces without claiming new implementation is complete.
