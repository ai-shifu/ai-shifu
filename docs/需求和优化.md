# 需求和优化

## 背景

- 主题：运营管理 -> 课程详情 -> 追问数据页
- 当前分支：`feat/operator-course-followup-page`
- 当前重点：
  - 先保证追问详情中的 source 回溯尽量符合历史事实
  - 把后续性能优化和继续排查项沉淀下来，方便后面定期回捞

## 状态说明

- `已完成`：当前分支已经落地，后续只需回归验证或观察
- `部分完成`：已经处理了一部分，但仍有残留范围需要继续统一
- `待处理`：尚未开始，后续回捞时需要优先确认是否仍有价值

## 已落地修复

### 1. 追问详情的 element source 改为按追问时间回溯历史快照

- 状态：`已完成`

- 问题：之前直接读取最新 active 的 anchor element，历史追问可能读到后来 reload 后的新内容
- 当前口径：
  - 已知 `anchor_element_bid`
  - 按 `created_at <= 当前追问时间` 回查最近一条未删除的 element 快照
  - 不再依赖“当前是否 active”判断历史 source

### 2. 追问详情的 block fallback 改为允许命中历史 superseded teacher block

- 状态：`已完成`

- 问题：之前 fallback 只查 `status == 1` 的 teacher block，历史上被 supersede 的原始输出会查不到
- 当前口径：
  - 同 `progress_record_bid`
  - 同 `position`
  - `role = teacher`
  - `type in (MDINTERACTION, MDCONTENT)`
  - 时间早于当前 ask
  - 允许命中历史行，不再只看 active 行

### 3. 翻页组件位置与运营其他页面保持一致

- 状态：`已完成`

- 已处理：
  - 追问数据页分页组件已改为与运营其他列表页一致的靠右对齐方式
  - 对齐参数与课程详情页、运营用户页保持一致

### 4. 无章结构课程时，列表与筛选文案按“节”展示

- 状态：`部分完成`

- 已处理：
  - 当当前列表数据不存在真实章 / 节两层结构时：
    - 列表列名从“章节”切换为“节”
    - 筛选文案从“章/节搜索”切换为“节搜索”
    - placeholder 同步切换为“输入节标题”
- 仍待继续统一：
  - 详情字段名
  - 相关空态或提示文案
  - 其他后续可能新增的追问页派生视图

## 待继续的优化项

### 3. 追问列表接口的 SQL 化过滤与分页

- 状态：`待处理`

- 问题：
  - 现在列表接口会先把课程内所有追问全量拉出，再做 Python 侧过滤、排序、分页
  - 追问量变大后，接口开销和页面体感都会明显变差

- 当前实现风险：
  - 接口耗时随课程追问量线性增长
  - 用户信息也会跟着全量补齐，增加额外开销
  - 后续如果再扩筛选项，Python 侧逻辑会继续膨胀

## 后续建议方案

### Phase 1: 最小可行优化

- 把这些过滤条件下推到 SQL：
  - `keyword`
  - `chapter_keyword`
  - `start_time`
  - `end_time`
- 列表查询改为：
  - 一次 `count(*)`
  - 一次当前页数据查询
- 用户信息改为：
  - 只对当前页 `user_bid` 做批量补齐

### Phase 2: 结构优化

- 如果 `chapter_keyword` 还需要兼容章/节双层模糊匹配：
  - 可以先在 outline 上做 BID 集合预筛
  - 再把目标 `outline_item_bid` 集合下推到 follow-up 主查询
- 如果后面还要补更多统计卡片：
  - 可以拆出 summary query，避免列表和 summary 共用同一套全量内存逻辑

## 回捞时优先检查

- 课程追问量较大时，接口耗时是否明显上升
- summary 与列表 total 是否仍然共用全量内存计算
- `keyword` 是否仍依赖全量 user map 才能过滤
- 是否已经存在可复用的 outline / user / pagination 查询 helper

## 回捞建议顺序

1. 先收集真实慢查询或页面体感数据
2. 再做 SQL 下推的最小改造
3. 保持接口返回结构不变，优先减少回归范围
4. 每次优化后补一组带分页和筛选的后端测试

## 页面产品优化项

### 总体判断

- 当前页面的大结构合理：
  - 概览卡片
  - 筛选区
  - 列表区
  - 详情抽屉
- 当前页面的主要问题不在大框架，而在语义容易被误解：
  - source 缺失态表达过弱
  - 追问轮次定义不够清晰
  - 章 / 节字段在不同课程结构下容易让运营困惑
  - 详情中“原始输出 / 本次追问 / 系统回答”的先后关系不够一眼看懂

### 必须改

#### 1. source 缺失态要明确表达

- 状态：`已完成`

- 当前问题：
  - 直接显示 `--` 信息量过低
  - 运营无法区分“确实没有”还是“历史数据暂时无法定位”
- 建议：
  - 改为：
    - `暂无法定位对应原始输出`
    - 或 `历史数据未记录对应原始输出`
- 目标：
  - 降低误解
  - 避免用户把 source 缺失理解为系统异常或空数据

#### 2. 明确“追问轮次”的定义

- 状态：`已完成`

- 当前问题：
  - `第 N 次` 很容易被理解成“对同一条原始输出的第 N 次连续追问”
  - 但当前实现实际是“同一学习记录中的追问时间顺序”
- 建议：
  - 在列表列头或详情字段旁增加 tooltip / 说明文案
  - 文案建议：
    - `按同一学习记录中的追问时间顺序排序`
- 目标：
  - 让运营正确理解轮次口径

#### 3. 章节 / 节语义继续统一

- 状态：`部分完成`

- 当前问题：
  - 有的课程存在章 + 节两层结构
  - 有的课程只有节，没有章
  - 如果页面文案写死“章/节”或“章节”，会造成理解偏差
- 建议：
  - 继续统一以下位置的口径：
    - 列表列名
    - 筛选文案
    - 详情字段名
    - 空态文案
  - 当课程无章结构时，优先统一显示为“节”
- 目标：
  - 保持整页语义一致
- 备注：
  - 当前分支已完成列表列名和筛选文案的动态切换
  - 详情字段名和其他文案仍需后续补齐

#### 4. 详情中三类内容的关系要更清楚

- 状态：`已完成`

- 当前问题：
  - 用户需要自己推断：
    - 被追问的原始输出
    - 当前追问
    - 系统回答
  - 尤其 source 缺失时，更容易困惑
- 建议：
  - 在标题和布局顺序上更明确地区分这三者
  - 推荐顺序：
    - `被追问的原始输出`
    - `本次追问`
    - `系统回答`
- 目标：
  - 让运营快速理解“因 -> 问 -> 答”的关系

### 建议改

#### 5. 概览区增加“基于当前筛选结果”提示

- 状态：`已完成`

- 当前问题：
  - 虽然统计逻辑已经是按当前筛选条件计算，但页面感知不够强
- 建议：
  - 在概览区右上角或筛选区下方增加轻提示：
    - `以下统计基于当前筛选条件`
- 目标：
  - 减少统计口径疑问

#### 6. 详情抽屉的信息顺序再优化

- 状态：`已完成`

- 建议顺序：
  1. 用户
  2. 节 / 章节
  3. 追问时间
  4. 追问轮次
  5. 被追问的原始输出
  6. 本次追问
  7. 系统回答
  8. 上下文时间线
- 目标：
  - 更符合“先定位对象，再理解内容，再回溯上下文”的阅读路径

#### 7. 列表中的“追问内容”列更像详情入口

- 状态：`已完成`

- 当前问题：
  - 目前虽然可点，但点击感知不够强
- 建议：
  - 强化 hover 或点击态
  - 让运营更自然地把追问内容理解为主入口，而不是只点“查看详情”
- 目标：
  - 降低操作成本

#### 8. 概览卡片与列表的视觉关系更紧一点

- 状态：`部分完成`

- 当前问题：
  - 概览是概览，列表是列表，二者关系偏弱
- 建议：
  - 用轻文案或标题增强两者关联
  - 例如把筛选区标题明确成：
    - `追问列表筛选`
- 目标：
  - 让页面更像一整套数据分析界面

#### 9. 时间线中更强调“当前选中轮次”

- 状态：`已完成`

- 当前问题：
  - 当前轮次已有高亮，但仍可以再增强识别
- 建议：
  - 提升高亮层级，或增加更明显的视觉标记
- 目标：
  - 在多轮上下文中更快定位当前问题

- 可选筛选值：
  - 全部

#### 11. 在详情中增加轻量“数据说明”

- 状态：`待处理`

- 注意：
  - 文案应面向运营，不要太技术化

#### 12. 支持从时间线反向理解列表上下文

- 状态：`待处理`

- 目标：
  - 帮助运营在长时间线场景中更好理解这条追问在整个学习过程中的位置
- 备注：
  - 不是第一优先级，可后续迭代

#### 13. 给追问增加轻量类型标识

- 状态：`待处理`

- 可能类型：
  - 文本追问
  - 图片追问
  - 听课模式追问
- 目标：
  - 帮助运营更快判断问题类型和排查路径

#### 14. 再做一轮“课程详情 / 用户页 / 追问页”的体验一致性收口

- 状态：`待处理`

- 结论：
  - 当前追问数据页已经可用，没有新的阻断性问题
  - 这一项主要是视觉和信息层级的一致性优化，可放到下一轮统一处理

- 本轮自查发现的残留点：
  - 追问列表卡片头部样式，和课程详情页“课程用户”区块的分区头风格还不完全一致
  - 追问页概览卡片的体量、字号、阴影仍略重于课程详情页概览卡片
  - 详情抽屉里的“轮次说明”仍是块级提示，和列表页里更轻的辅助说明不够统一

- 下次优化建议：
  - 统一区块标题、辅助说明、卡片阴影和留白节奏
  - 让课程详情页、课程用户区、追问页三者更像同一套运营后台页面
  - 只做体验收口，不改当前数据逻辑，避免扩大回归范围

### 页面优化的推荐优先级

1. source 缺失态文案
2. 追问轮次定义说明
3. 详情三段内容标题与顺序优化
4. 概览区补“基于当前筛选结果”提示
5. 列表主入口点击感增强

### 页面优化的实施原则

- 不大改页面骨架，优先降低歧义
- 先解决“看不懂”，再解决“更好看”
- 先保证运营判断成本低，再继续追求交互精致度
